QR Code Champion Roles & Responsibilities

QR Code Champion Roles & Responsibilities

Main activities

  • Define site activities, including KPI governance, to secure QR-Code process to feed digital logbook incl. a high data accuracy
  • SPOC for QR-Code related questions in their respective site from customers, partners, suppliers and Schneider Electric
  • Industrialize creation of the QR Codes on the lines (Purchase of printer, definition of the location with industrial engineer, …) + define process of upload of the documents by respective Site

    • Daily basis

  • Contact for QR-code questions / support / escalation in their respective site

    • On regular basis

  • Provide Training and Refresher for concerned people in their respective site
  • Participate to monthly calls with the QR-code community and other webinars (M2C, EPC, E2E) to understand current and future projects and process related information.
  • Assure the communication in his plant for the relevant information
  • Setup / Update settings in QR-code related software used by plant (M2C, EPC, …)
  • Support root cause analysis in case of customer complaint related to QR-code and digital logbook
  • Propose through the ticketing system enhancements of the tool to improve efficiency

    • Being a QR Code Champion is not a full-time job, but it is a very important role that requires flexibility. Depending on project status, process robustness and process escalations the load has to be adapted very flexibly.

  • Workload related to new projects depends on the nature of the project: limited period
  • Responsible of the QR-Code process in their respective site
  • Project Management competency, Business knowledge & Digital Savvy profile along with connected offer basic knowledge are required
  • QR Code Champion back-up is a must for a sustainable QR-code performance and a robust process

RAC definition

There must be only one accountable specified for each task or deliverable

R = Responsible. Those who do the work to complete the task

A = Accountable (or final approving authority). The one ultimately answerable for the correct and thorough completion of the deliverable or task, the one who ensures the prerequisites of the task are met and who delegates the work to those responsible

C = Consulted (sometimes contributor). Those whose opinions are sought, typically subject-matter experts

How can you create an asset in M2C V2.0 (Cloud Synch)

How can you create an asset in M2C V2.0 (Cloud Synch)

It is an automated method to upload bulk/ single documents into M2C basis the documents that are placed in the cloud shared folder . Two applications are there:

    • If asset is already created: Use this functionality to mass upload documents associated with the asset
    • If asset is not created: By placing the files for any given asset in the shared folder, M2C will automatically create the asset and tag the documents under the asset using Cloud Synch

High level user flow

1. Go to shared folder path

3. Place your documents in either the equipment/ product folder (To do folder)

2. Rename the documents of the asset in the defined file format

4. M2C will process 40 do / 45 min and documents will reflect in M2C application

File Format

If Equipment:

File format: <Project Reference> #<Generic Comm Ref.> # <Serial Number> # <DocType># <Language>

If Product*:

File Format: <Product Name>_<Generic Comm Ref.>_<Serial Number>_<DocType>_<Language>

Language Format: EN for English . #Other formats :DE/RU/ES/TR/PT/AR/HY/ZH/FR

* Product Name – Field not mandatory

# To add any other language format please raise a ticket or contact the M2C super admin

Some key instructions to follow while using cloud sync

Some key instructions to follow while using cloud sync

  • The document name and Com ref number must match exactly and be listed in the plant settings.
  • The document type (example, Shipment Certificate / RTR / etc) must be available for the plant and must match the document type that is configured in the plant settings.
  • Cloud Sync checks the document type against the list of document types that are configured for the plant. It is allowed to upload documents that are not configured for a product in plant.
  • The serial number must be unique and cannot be used by any other plant.
  • The document must have one of the allowed file extensions. Other file extensions are not allowed
  • If a user tries to upload a document that already exists in ALM, a new version of the document will be created.
  • The document file size must be within the 100 MB in M2C.
  • Cloud Sync BOT will sync documents one by one. In each 45-minute cycle, 40 documents will be synched per plant, depending on whether the documents are for equipment or products.

    • 9. For new plants, the Cloud Sync folder structure must exist and be accessible to SESA or SESI users

Possible Parent - Child Hierarchy

  • KIOSK = SWB, CUB, DEVICE,COMPONENT
  • KIOSK ,SWB = CUB, DEVICE,COMPONENT
  • KIOSK ,SWB ,CUB = DEVICE,COMPONENT
  • KIOSK ,SWB ,CUB , DEVICE = COMPONENT

If the parent is a KIOSK, we have four possible hierarchy types that can be added. However, if the parent is a COMPONENT, no further hierarchy can be created, as it is the last layer and cannot act as a parent to any other hierarchy type. This rule applies similarly in all cases.
